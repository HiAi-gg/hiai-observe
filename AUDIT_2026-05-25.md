## CRITICAL — Немедленные исправления

### 1. WebSocket без аутентификации (Security: 3/10)
- **Файл:** `src/api/logs-ws.ts:14-83`
- `/ws/logs` — нулевая проверка API key. Любой, кто достучался до порта, видит ВСЕ логи контейнеров.
- `subscribe_all` (line 59) — подписка на все логи без ограничений.
- Нет rate-limiting на WS-сообщения.
- **Фикс:** API key через query param или first-message auth, scope по projectId.

### 2. Cross-Project Authorization отсутствует (Security: 4/10)
- **Файл:** `src/middleware/auth.ts:94-108`
- Auth-мидлвар возвращает `projectId`, но ни один роут не проверяет `callerProjectId === resourceProjectId`.
- Любой валидный API key имеет полный доступ ко ВСЕМ проектам.
- Затронуты: projects.ts (rotate-key, delete), export.ts (все 3 endpoints), alerts.ts, notifications.ts, sourcemaps.ts.
- **Фикс:** Проверка projectId в каждом роуте.

### 3. Email-нотификер — заглушка (Backend: 5/10)
- **Файл:** `src/alerts/notifiers/email.ts:138-163`
- `Bun.connect()` открывает TCP-сокет, НЕ делает SMTP-диалог, сразу `socket.end()` и возвращает `{ ok: true }`.
- Email-уведомления молча не отправляются.
- **Фикс:** Реализовать SMTP через nodemailer или handshaked SMTP.

### 4. N+1 queries в status-page-html (Backend: 5/10)
- **Файл:** `src/api/status-page-html.ts:66-98`
- На каждый монитор: 3 запроса (latest check, uptime stats, all checks). 20 мониторов = 60 запросов на одну страницу.
- **Фикс:** Batch query с JOIN или `DISTINCT ON`.

### 5. 3 отдельных Redis-подключения (Backend: 6/10)
- `src/store/redis.ts` — основной singleton
- `src/alerts/dedup.ts:13-22` — своё подключение
- `src/store/log-pubsub.ts:9-10` — publisher + subscriber (ещё 2 подключения)
- Итого: 4 Redis-соединения вместо 1.
- **Фикс:** Использовать общий singleton из `store/redis.ts`.

## HIGH — Серьёзные проблемы

### 6. Tripled auth logic (Backend: 6/10)
- `src/middleware/auth.ts` — основной
- `src/api/sentry-ingest.ts:9-48` — `resolveApiKey()` — дубликат #1
- `src/api/otlp.ts:113-133` — `extractApiKey()` — дубликат #2
- Каждый имеет разную логику парсинга и семантику ошибок.
- **Фикс:** Один модуль `lib/auth.ts` для всех.

### 7. Docker runs as root (Docker: 4/10)
- **Файл:** `Dockerfile` — нет `USER` директивы.
- Docker socket смонтирован напрямую (docker-compose.yml:62) — компрометация контейнера = полный доступ к хосту.
- **Фикс:** `USER bun`, Docker socket proxy (tecnativa/docker-socket-proxy).

### 8. No backup strategy (Production: 2/10)
- Нет pg_dump скриптов, нет документации, нет cron-задач.
- `scripts/reset-demo.ts` удаляет ВСЁ без подтверждения.
- **Фикс:** Скрипт backup, документация, cron.

### 9. Broken wildcard Redis subscription (Architecture: 5/10)
- **Файл:** `src/store/log-pubsub.ts:101`
- `sub.subscribe("logs:*")` подписывается на ЛИТЕРАЛЬНУЮ строку `"logs:*"`, а не на паттерн.
- "Subscribe all logs" в WebSocket полностью не работает.
- **Фикс:** Использовать `sub.psubscribe("logs:*")` (pattern subscribe).

### 10. Type Safety — 15 unsafe casts (Backend: 5/10)
- `sentry-ingest.ts:81` — `body as SentryEvent`
- `otlp.ts:50,89,129` — multiple unsafe casts
- `alerts.ts:52-54,119-120` — double casts `as unknown as`
- `logs-ws.ts:27,41,74` — monkey-patching `ws._clientId`
- `rules-engine.ts:134-137` — `db.execute()` returns unknown
- **Фикс:** Zod-валидация на ingestion, типизированные store-запросы.

## MEDIUM — Важные улучшения

### 11. Test Coverage: 4.6/10
- 12/52 файлов покрыто тестами.
- 15 API route handlers — **ноль** route-level тестов.
- 4 middleware файла — **ноль** тестов.
- Весь alert pipeline (dedup, dispatcher, 4 notifiers) — **ноль** тестов.
- Integration/E2E тесты есть, но НИКОГДА не запускаются в CI (skipIf(!INTEGRATION=1)).
- CI: нет lint, нет coverage, нет frontend build, Docker check фейковый (`|| true`).

### 12. Worker Architecture: 4/10
- 5 воркеров в одном процессе, без supervision.
- Crash одного воркера = падение API.
- Нет health checks воркеров, нет circuit breaker.
- Все `setInterval` стартуют одновременно при запуске (нет jitter).
- Shared connection pool (20) между 5 воркерами и API = contention.

### 13. Scaling: 4/10
- Single process, shared event loop.
- <50 req/sec — честное ограничание из README.
- Нет read replicas, нет clustering.
- `setInterval` воркеры не cluster-aware — двойное выполнение при масштабировании.

### 14. Input Validation gaps
- `sentry-ingest.ts:89,119` — `body: t.Any()`, без лимита размера.
- `otlp.ts:69,107` — `t.Array(t.Any())`, безграничные массивы.
- `sourcemaps.ts:40` — upload без лимита размера файла.
- `logs.ts:36` — `limit` без верхнего предела (`limit=999999999`).

### 15. Metrics-as-traces anti-pattern (Architecture: 5/10)
- **Файл:** `src/ingestion/otlp-parser.ts:121-177`
- OTLP metrics записываются в таблицу `traces` с синтетическими ID.
- Разная кардинальность метрик и спанов = table bloat и incorrect aggregations.
- **Фикс:** Отдельная таблица `metrics`.

### 16. Email notifier stub + Recovery alert never fires
- `recovery` alert type объявлен в `rules-engine.ts:20` но никогда не evaluates.
- Slack не в `VALID_CHANNELS` (notifications.ts:16), но `dispatcher.ts:122` его обрабатывает.
- Certificate expiry detection — заглушка (uptime-worker.ts:32-40).
- Network rate fields всегда 0 (infra-worker.ts:17-35).

- Чистое разделение: api/, store/, middleware/, alerts/, ingestion/, mastra/, monitoring/, workers/
- Каждый API route — независимый Elysia plugin
- Store layer абстрагирует DB от routes
- Sentry parser + OTLP parser отдельно от storage
- Graceful shutdown (index.ts:92-116) — останавливает все 5 воркеров, Redis, PG

### API Design: 7/10
- Консистентная пагинация: `{ data, total, limit, offset }`
- Правильные HTTP status codes (201, 404, 401, 400)
- TypeBox валидация на большинстве endpoints
- Rate limiting с заголовками (X-RateLimit-*, Retry-After)
- Prometheus /metrics endpoint

### SQL Injection: 9/10
- Drizzle ORM с tagged template literals — все значения параметризованы.
- Никакой string interpolation в SQL.
- `sql.raw()` только на hardcoded table names (retention.ts).

### CORS: 8/10
- `cors({ origin: process.env.CORS_ORIGIN || false })` — secure default (нет wildcard).
- Опасность только в operator misconfiguration.

### Dependency Security: 8/10
- Все зависимости на свежих версиях.
- Нет известных критических уязвимостей.

### Docker Setup: 8/10 (prod)

- `depends_on` с `condition: service_healthy`.
- Security headers через Caddy.

### Documentation: 8/10

### Mastra Differentiator: 7/10

### Operational Readiness: 8/10
- Health check: PG + Redis, "ok"/"degraded"/"error", memory stats, uptime.
- Graceful shutdown: SIGTERM/SIGINT, all workers, Redis, PG.
- Prometheus metrics: request count, error count, latency histogram (11 buckets).
- Auth cache: 60s TTL, 1000 entries, avoid DB hits per request.
